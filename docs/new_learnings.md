### 5. Core Web Vitals 2026 — Clarification INP et donnees terrain

**Precision importante** : Les seuils CWV **n'ont pas change** (LCP ≤2.5s, INP ≤200ms, CLS ≤0.1). Ce qui a change :

1. **INP remplace FID** : INP mesure TOUTES les interactions pendant la visite, pas juste la premiere. Revele les problemes avec widgets chat, formulaires, navigation
2. **Donnees terrain (CrUX) > Lighthouse** : Google evalue au 75e percentile des vrais utilisateurs, pas les scores labo
3. **Approche par templates** : corriger le modele de page (ex: reserver.html) impacte toutes les URLs, plus efficace que corriger page par page
4. **Fenetre de validation** : attendre 28 jours avant de declarer un CWV "passe" (fenetre d'agregation Google)

**CWV = tiebreaker, pas filtre principal** : Google precise que les CWV servent surtout quand la pertinence est equivalente entre deux resultats. Pour un barbershop local, les avis et la GBP pesent bien plus que le score Lighthouse. Mais un site lent reste penalise indirectement (taux de rebond, conversions).

**Action BarberClub** : Le site BarberClub (HTML vanilla, images optimisees) a naturellement de bons CWV. Verifier que les pages reserver.html passent INP <200ms avec les interactions formulaire complexes (selectors, date pickers).

### 7. CSS content-visibility — gain de 40% en rendering pour les pages longues

**Technique CSS applicable au site vitrine** : La propriete `content-visibility: auto` permet au navigateur de sauter le rendu des elements hors ecran. Combinee avec `contain-intrinsic-size`, elle offre un **gain de ~40% en rendering et painting** sur les pages avec beaucoup de contenu.

**Implementation** :
```css
.section-offscreen {
  content-visibility: auto;
  contain-intrinsic-size: auto 500px;
}
```

**Fonctionnement** :
- Le navigateur ne rend pas les sections hors viewport → chargement initial plus rapide
- `contain-intrinsic-size` donne une taille placeholder pour eviter les sauts de scrollbar
- Le mot-cle `auto` retient la taille rendue apres le premier affichage
- Support : Chrome, Edge, Firefox (depuis v125). Safari : support partiel

**Attention** : Peut causer des sauts de scrollbar quand les sections entrent dans le viewport. A tester soigneusement sur les pages longues (reserver.html ~3900 lignes, prestations.html).

**Impact BarberClub** : Applicable sur les pages avec beaucoup de sections (galerie, prestations, contact avec carte). Pourrait aider a maintenir le LCP < 2.0s (nouveau seuil Google mars 2026). A tester en priorite sur les pages les plus lourdes.

- Seuils "Needs improvement" INP : 201-500ms ; "Poor" : > 500ms

**Checklist INP optimisation** :
- Éliminer le JS long-task (> 50ms) au chargement : chunker les event handlers lourds
- `requestIdleCallback` pour travaux non-critiques
- Ne pas bloquer le main thread avec des `JSON.parse` massifs ou des `forEach` sur grosses collections
- Tester avec `PerformanceObserver({ type: 'event' })` en dev
- Field data via CrUX (28 derniers jours) > lab data

**Impact BarberClub** : Le site est déjà en HTML/CSS/JS vanilla (pas de React sur le site vitrine), donc INP théoriquement bon. Points de vigilance :
- **reserver.html** : 3900 lignes avec gestion de state en JS vanilla pour le flow 4 étapes. Vérifier que les `click` handlers (sélection barber, service, créneau) n'excèdent pas 50ms sur mobile bas de gamme
- **Carousels barbers** (meylan/grenoble index.html) : les animations doivent être en CSS pure, pas en `setInterval` JS
- **phone-country.js** (sélecteur avec recherche) : mesurer l'INP du `input` event sur la recherche (filtrage de ~240 pays)
- **Service Worker** : le cache network-first est bon pour le retour, mais pour un premier chargement le LCP dépend du backend Railway. Ajouter un `preconnect` vers Railway en tête de page

Test recommandé : PageSpeed Insights sur `reserver.html` avec throttling "Slow 4G" → objectif INP < 200ms sur clics sélection créneau.

## Veille du 2026-04-27 — Fiabilité SMS Brevo (zéro silent fail)

Contexte : credits Brevo Grenoble vidés à 18:33 le 27/04. Notre queue a marqué 6 SMS "sent" alors que Brevo les a rejetés silencieusement (API renvoie 200 OK même quand le SMS est ensuite "Rejeté"). Détecté uniquement via screenshot Brevo logs par Nino.

| Score | Source | Trouvaille | URL |
|-------|--------|-----------|-----|
| 5/5 | Brevo Docs | Webhooks transactionnels SMS supportent 11 events dont `delivered`, `rejected`, `hardBounce`, `softBounce`, `blacklisted`, `accepted` — payload contient `messageId`, `to`, `msg_status`, `error_code`, `description` | https://developers.brevo.com/docs/transactional-webhooks |
| 5/5 | Brevo API | Création webhook via `POST /v3/webhooks` avec `{type:"transactional", channel:"sms", events:[...], url:"..."}` — renvoie `{id}` à stocker | https://developers.brevo.com/reference/create-webhook |
| 4/5 | Brevo Docs | `GET /v3/account` retourne `plan[]` avec `{type:"sms", credits:N}` — déjà utilisé dans `routes/admin/notifications.js:172` mais pas pour check pré-envoi | https://developers.brevo.com/reference/getaccount |
| 4/5 | Brevo Docs | `GET /v3/transactionalSMS/statistics/events` permet d'auditer a posteriori les events rejected/delivered sur N jours | https://developers.brevo.com/docs/transactional-sms-endpoints |
| 3/5 | Brevo Help | Logs SMS transactionnels montrent statut "Rejeté" avec raison — accessible dans dashboard `app.brevo.com/transactional/sms/logs` (source unique de vérité delivery) | https://help.brevo.com/hc/en-us/articles/4445508727698 |

### Insight clef

**Brevo SMS est asynchrone** : l'API `POST /v3/transactionalSMS/send` renvoie 200 + `messageId` même quand le SMS sera ensuite rejeté par le carrier (crédits, blacklist, format). Notre queue ne peut PAS savoir si le SMS est délivré sans :
1. **Soit** un webhook qui reçoit `delivered`/`rejected` events de Brevo en push
2. **Soit** un poll de `GET /v3/transactionalSMS/statistics/events` pour cross-check les `messageId` enregistrés dans `notification_queue.provider_message_id` (déjà stocké, migration 047 !)

### Solution recommandée pour BarberClub

**Architecture en 3 couches de défense :**

1. **Pre-send credit gate** (rapide) : avant chaque `brevoSMS()`, lire le solde via `latestCredits[salonId]` (in-memory, mis à jour à chaque envoi via `remainingCredits` du payload). Si solde < seuil critique (ex: 5), bloquer l'envoi SMS et fallback email + alerte Discord. Seuil bas existe déjà (`BREVO_CREDIT_LOW_THRESHOLD: 50`) mais ne bloque pas, juste alerte.

2. **Webhook delivery confirmation** (la vraie solution) :
   - Endpoint public `POST /api/webhooks/brevo/sms` (sans auth, mais avec secret signature dans header ou IP whitelist)
   - À la création webhook Brevo : `events: ["delivered", "rejected", "hardBounce", "softBounce", "blacklisted"]`
   - Sur réception : `UPDATE notification_queue SET status='delivered'/'failed', last_error=description WHERE provider_message_id = $1`
   - Si `rejected` → re-queue automatiquement OU alerte Discord par client
   - Bonus : tracking ouverture/livraison réelle pour analytics

3. **Polling fallback** (filet de sécurité) : cron toutes les 15 min qui appelle `GET /v3/transactionalSMS/statistics/events?startDate&endDate&event=rejected` pour les SMS récents et reconcilie avec notre queue. Catche les webhooks ratés.

### Patterns SaaS comparables (référence)

Twilio, Vonage, MessageBird ont tous le même problème : API renvoie OK mais delivery réelle est async. Tous résolvent via webhook callback obligatoire en prod. Brevo est aligné avec cette pratique standard.
